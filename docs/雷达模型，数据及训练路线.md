# 受限条件下的雷达数据采集与小样本建模研究报告

## 执行摘要

在被测人数、场景和采集设备都受限的学生实验条件下，最重要的结论是：**不要一开始就把目标设成“端到端大模型吃全量原始信号”**。更稳妥的做法是同时保存四层信息：其一是尽可能上游的**复数基带/原始 ADC-IQ**；其二是稳定、易复现实验的**中间表征**，例如距离谱、微多普勒谱、距离-多普勒图、距离-角度图；其三是便于做学习和评估的**点云/检测/轨迹**；其四是对复现实验成败影响极大的**配置与同步元数据**。这样既能支持传统信号处理，也能支持后续迁移学习、少样本学习和图网络。citeturn35view0turn35view1turn36view0turn36view1turn37view0turn42view0

如果只能做一套最小可行系统，优先推荐带原始数据接口的 **60/64 GHz FMCW MIMO 板卡**，例如可参考 entity["company","Texas Instruments","semiconductor company"] 的 IWR6843 + DCA1000 路线，或同类 60 GHz 开发板；这类设备通常兼顾距离、速度和角度三维信息，且官方工具链较成熟。若你手上只有 entity["company","Infineon Technologies","semiconductor company"] 的 24 GHz 脉冲/低成本 FMCW 模块，也仍然可以完成存在检测、简单动作分类、手势识别或微多普勒研究，但不应把“高质量角度估计”或“全身三维姿态”作为首个目标。关于“雷达数据立方”“RD/RA/AD 响应”等术语和处理坐标系，entity["organization","MathWorks","software company"] 的官方文档可作为统一参考。citeturn41view1turn41view0turn35view6turn41view2turn35view1turn36view0turn36view1

在建模上，小样本条件下最稳的路线通常是：**传统信号处理 + 轻量特征工程基线** → **迁移学习微调** → **少样本/元学习**。只有当你的点云质量、追踪稳定性与时序长度都足够好时，再上 **PointNet++ / ST-GCN / PRGCN / 时序 Transformer** 之类更复杂的模型。对于“跨房间、跨设备、跨频段”这种明显存在域偏移的问题，**合成数据 + 领域自适应** 往往比单纯堆网络更有效。

## 采集数据项与最小配置

### 建议采集的核心数据项

下表按“**必须保存** / **强烈建议** / **有条件再保存**”组织。面向学生实验，最关键的原则是：**至少保留一条可回放、可重处理的链路**。即便你平时只训练 RD 图分类器，也应在一部分数据上保存原始 IQ，以便后续重做滤波、相位处理、角度估计和去静态杂波。citeturn35view0turn35view1turn36view0turn37view0

| 距离-角度图 RA / 角度-多普勒图 AD / RAD tensor | 强烈建议 | 至少 2 Rx，最好 MIMO | 对阵列通道做波束形成或 AoA 估计；多 Tx/多 Rx 先做虚拟阵列整形 | 起步建议 2 Rx 以上；更实用的是 3Tx4Rx 级别 MIMO | 角度估计至少需要 2 个接收天线；角度分辨率受接收通道数/孔径限制。 |

| 信号与实验元数据 | 必须 | 所有系统 | 单独保存 JSON/CSV：时间戳、chirp slope、带宽、采样率、PRF/帧率、Tx/Rx mask、天线排列、固件、房间/人物/动作标签、设备位姿 | 每个 trial 一个 metadata 文件，配置用 hash 固定 |  |

### 采集策略上的取舍建议

受限条件下，不建议把所有 trial 都以原始 IQ 长时间保存。更实际的做法是：

- **全量保存**：RD/RA/点云/轨迹/元数据。  
- **抽样保存**：每类、每人、每场景至少保留 **10%–20%** 的原始 IQ，用于后续重处理、相位研究和基线复现。  
- **每次 session 额外保存**：空场、静态站立、已知距离参考点、传感器重新摆位后的短标定片段。  

这是工程上的存储-复现折中。像 CARRADA 的 RAD tensor 压缩包就达到 176 GB，而 RADIal 直接保存原始 ADC 数据与多传感器日志；这说明“全量原始数据”在学生机器上通常并不经济，但“完全不保留原始数据”也会使后续方法受限。

## 采集参数与同步要求

### 各类雷达的起步配置建议

下面给出的不是唯一正确值，而是**在学生实验中足够保守、足以起步的工程起点**。它们的依据来自官方器件能力、公开数据集常见设置与分辨率公式。对 FMCW 来说，距离分辨率主要由带宽决定，速度分辨率与帧时长/相干处理时间相关，角度分辨率与接收通道数或虚拟阵列孔径相关。

| 硬件类型 | 更适合的任务 | 推荐起步值 | 天线与输出建议 | 不建议一开始做的任务 |
|---|---|---|---|---|
| 24 GHz 脉冲/脉冲多普勒模块 | 存在检测、移动/静止判别、简单速度谱、低速手势 | 重点记录 PRF、脉宽、每 CPI 脉冲数、采样率、匹配滤波系数；短时窗即可 | 常见 1Tx1Rx，优先保存脉冲栈、Doppler 谱、匹配滤波后距离序列 | 高质量角度估计、多人近距离分离、复杂姿态回归。 |
| 24 GHz 低成本 FMCW | 距离测量、1D 跟踪、简单双分类/多分类 | 若硬件支持，尽量开到更大带宽；完整记录 chirp slope、B、Fs、chirps/frame | 1Tx1Rx 或 2Tx2Rx 为主；若只有单通道，主存 RD 与微多普勒，不强行做角度学习 | 跨人群精细步态识别、稳定多人跟踪。 |
| 60/64 GHz FMCW MIMO 单板 | 动作识别、手势、基础定位、点云与轨迹、近距人体研究 | **B = 2–4 GHz**、ADC **4–8 Msps**、**128–256 samples/chirp**、**32–128 chirps/frame**；人体动作 **10–20 fps**，手势 **20–60 fps**；trial 2–10 s | 推荐至少 **2 Rx**，更实用的是 **3Tx4Rx** 量级；若能出点云，则同时存 RD、RA、点云、track | 初学阶段直接做大规模 3D mesh / 多人复杂交互。|
| 77 GHz 高分辨率 / 级联 MIMO | 车辆与行人检测、较精的方位估计、研究级多传感器对齐 | 参考高分辨率公开集的原始 ADC + 日志方案，优先做检测/跟踪 | 可达 12Tx16Rx 级虚拟阵列；务必保存时间日志、外参、同步信号 | 学生项目首轮就追求全量多任务联合学习。citeturn21view0turn26view2 |

### 采样率、带宽、帧率与采集时长的实操规则

**带宽**：FMCW 的第一关键量是带宽，距离分辨率随带宽单调改善，近似满足 **ΔR = c/(2B)**。如果你研究的是近距人体动作、手势或床旁/桌旁行为，优先把带宽开到该板卡允许的较高水平，再在存储与处理量之间做平衡。TI 的入门 FMCW 教程用 4 GHz 带宽示例说明可达到厘米级距离分辨能力；IWR6843 官方规格也给出 60–64 GHz、4 GHz 连续带宽和 3Tx4Rx 通道配置。

**采样率**：ADC 采样率不是越高越好，而是要与 IF 带宽、可观测距离和设备链路一起设计。Infineon 的官方说明指出，ADC 采样率决定模拟输入被数字化的频率，更高采样率通常带来更完整的信号表示与更好的检测/分辨表现，但也增加数据量与处理负担。对学生项目，单板 FMCW 起步用 **4–8 Msps** 往往比“极限采样率”更平衡。

**帧率**：人体姿态/存在检测通常 **5–10 fps** 足够；步态、日常动作与轻微交互建议 **10–20 fps**；动态手势与细粒度微动作建议 **20–60 fps**。这与公开数据集中人体 pose/行为任务常见的 10–20 fps 级别设置一致，而更快手势任务则依赖更高时间密度。HIBER 使用 20 fps，HuPR 使用与视频同步的雷达序列做人姿估计，Pantomime 面向动态手势序列。

**采集时长**：  
- 离散动作分类：**2–4 s**/trial。  
- 日常动作、步态、转身、坐立：**5–10 s**/trial。  
- 连续监测与分割：单段 **30–120 s**，但必须保留动作边界。  
- 空场/背景：每次 session 前后各 **10–30 s**。  
这些值是工程推荐，核心目标是让一个窗口内既能覆盖完整动作，也不过度拉长到引入过多无关静态背景。

### 同步要求

**单雷达单机**：  
最少要做到每个 frame/chirp 都有主机时间戳、配置文件版本号和 sensor pose 记录。若使用 TI 多板卡或外部相机，时间戳必须从“录制开始时间”提升为“每帧/每事件时间戳”。RADIal 的官方说明专门保存了每个传感器事件的日志文件，这比只记录序列开始时间更可靠。

**雷达 + 相机 / IMU / LiDAR**：  
如果只靠软件同时发 start 命令，TI 文档指出它只能做到**粗同步，精度约 1 ms**，且时间长了会因各设备时钟差异而漂移。硬件触发方案通过单一触发源给所有传感器送 SYNC_IN，时间一致性明显更好。对于学生项目，至少要做到：  
- 统一时基；  
- 保存各传感器逐帧时间戳；  
- 每个 sequence 开头安排一个明显的“同步动作”（例如快速挥手、拍掌、转身）；  
- 多雷达时优先硬件触发。 

**容差建议**：  
- 手势/微动作：尽量做到 **< 5 ms**。  
- 全身动作与一般 pose：**< 半个雷达 frame interval** 通常可接受。  
- 多雷达联合 AoA / 级联研究：尽量接近**硬件同步**。  
这些是工程性建议；证据基础是 TI 官方对软件同步的 ±1 ms 粗同步定义，以及公开多传感器数据集都显式依赖 synchronized recordings。

## 小样本模型路线比较

### 候选模型总表

| 模型类别 | 适用场景 | 推荐架构 | 典型输入格式 | 常用损失 | 主要优点 | 主要缺点 | 推荐训练策略与起步超参 | 代表依据 |
|---|---|---|---|---|---|---|---|---|
| 传统信号处理 + 特征工程 | 样本极少、需要解释性、课程项目首个基线 | STFT/CWT + handcrafted features + SVM / RF / XGBoost / HMM | 微多普勒谱、RD 图统计量、范围轨迹、峰值序列 | hinge / logloss | 对小数据最稳；解释性强；调试快 | 上限较低；迁移性差；需要人工设计特征 | 先做去静态、归一化、峰值跟踪；特征用 Doppler centroid、bandwidth、零多普勒能量比、range variance、周期性；SVM 先试 RBF，`C=1~10`，`gamma=scale` | 早期人类活动 micro-Doppler + SVM 文献与后续深度学习对比证明，手工特征基线仍有必要。 |
| 迁移学习 | 已有少量标签，但不足以从零训练 | ResNet-18 / ResNet-34 / EfficientNet-B0；对雷达图像做微调 | `1×H×W` 或 `2/3×H×W`，可用 `log-mag`、`phase`、`real-imag` | 交叉熵 | 训练稳定；对 1k–5k clip 很有效 | 可能把雷达当“伪图像”，物理归纳较弱 | 优先用 ImageNet 预训练；若有条件，用雷达专用合成预训练更好；前 5–10 epoch 冻结 stem，随后全量微调；`lr=1e-4~3e-4`，`batch=16~32`，`wd=1e-4`，`dropout=0.2~0.5` | ResNet 原始工作与雷达微多普勒迁移学习文献都表明，小样本下预训练显著优于从零训练；合成 micro-Doppler 预训练优于直接从光学图像迁移。 |
| 少样本学习 | 新增少量类别、新用户自定义手势、样本按类极少 | Prototypical Networks、Siamese、Relation Network | 短时谱图 patch、短 RD clip、点云片段 embedding | 原型损失 / contrastive / triplet | 对 1-shot / 5-shot 场景友好；新增类别成本低 | 依赖 episode 设计；基类与新类差异过大时不稳 | 用 5-way 1-shot/5-shot episodic 训练；backbone 先用轻量 CNN/TCN；margin 先试 `0.2~0.5`；每个 episode 中做 subject-balanced 采样 | ProtoNet 与 Triplet 网络是经典小样本度量学习方法，雷达手势工作已证明其能适配新用户和新上下文。 |
| 元学习 | 任务很多但每个任务数据都少；跨人/跨场景快速适配 | MAML / FOMAML / ProtoMAML | 与少样本类似，常用短 clip 或 embedding | inner/outer cross-entropy | 适合“快速适配” | 算法复杂，训练不稳，算力开销较大 | 仅当你有“多任务结构”时再用；inner loop 1–5 step；inner lr 小，outer lr 更小；优先用 FOMAML 降成本 | MAML 的目标正是让模型在极少梯度步内适配新任务；雷达手势元学习论文已验证快适配价值。 |
| 生成模型 / 合成数据 / 领域自适应 | 真实数据少、跨房间或跨频段差异大、难标注 | 物理仿真 + GAN/DCGAN/ACGAN；source-pretrain + DANN/CORAL | 合成 micro-Doppler、合成 RD、模拟点云、未标注目标域数据 | 对抗损失 + 交叉熵；领域对齐损失 | 适合扩增和缓解 sim-to-real / cross-frequency gap | 生成质量与域偏移难控；训练流程更复杂 | 先用物理先验或 mocap/simulator 生成粗数据，再少量真实数据微调；目标域无标签时，用 DANN/CORAL；前期 synthetic:real 可设 1:1，后期逐步提高真实样本权重 | 生成式 micro-Doppler 扩增、跨频段对抗训练与物理仿真到真实的 radar sim-to-real  |
| 时序模型 | 连续动作、动作边界分割、长序列行为分析 | CNN+BiLSTM、TCN、轻量 Transformer、状态空间模型 | `[T,H,W]` RD/RA clip；`[T,N,F]` 轨迹/点云特征 | clip-level 交叉熵；分割任务可用 frame-level CE | 能显式建模时间依赖；对相似动作更有帮助 | 更容易过拟合；对同步质量敏感 | 若数据 < 2k clip，优先 TCN 或 CNN+BiLSTM；窗口 16–64 帧；重叠 50–75%；`lr=1e-3` 从零训，`1e-4` 微调 | TCN 在时序建模上高效，RadHAR 和后续工作都使用 CNN/BiLSTM 或滑窗时序结构处理稀疏点云/雷达 clip。 |
| 图神经网络 / 点云网络 | 已有稳定点云、骨架或多点轨迹；人体姿态或细粒度动作 | PointNet++、Light-PointNet、ST-GCN、PRGCN、MMPoint-GNN | 点云序列、节点为点/关节、边为时空邻接 | 分类用 CE；姿态用 heatmap MSE / keypoint regression loss | 更贴近几何结构；适合 pose 与 point cloud HAR | 对点云质量极其敏感；稀疏雷达点云容易让模型失稳 | 没有稳定点云时不要硬上 GNN；先做点云去噪与 track 修复；节点特征至少含 xyz、doppler、power、angle；点数不足时先做 voxel/稀疏补偿 | PointNet++ 适合点集合层级特征；ST-GCN 适合骨架图；HuPR 的 PRGCN 直接用于 radar pose refinement；雷达点云 HAR 已有 MMPoint-GNN、Light-PointNet  |

### 训练策略的具体建议

对于**输入格式**，我更建议按“任务”而不是按“模型潮流”来定：

- **手势、动作分类**：优先 `log-magnitude RD`，并尝试增加 `phase` 或 `real/imag` 通道；2024 年关于 micro-Doppler 输入格式的研究明确指出，相位并非冗余信息。
- **连续行为 / 动作边界**：优先 `RD/RA clip + TCN/BiLSTM`；当帧率不高且标签稀疏时，TCN 往往比纯 Transformer 更稳。
- **点云/姿态**：只有在点云质量稳定、轨迹不过度断裂时，才切换到 PointNet++ / ST-GCN / PRGCN。否则宁可回退到 RD/RA。

对于**预训练/微调**，小样本时优先级通常是：  
**自监督或迁移预训练 > 从零训练**。对 radar 图像类输入，可从 ImageNet backbone 起步；对点云类输入，可用 MAE/point cloud masked autoencoder 先做无标签预训练；对检测类 radar perception，可用 radar-specific contrastive pretraining。

对于**数据增强**，建议只做**物理上合理**的增强：  
- 时间平移、时窗抖动；  
- 轻微 Doppler 平移；  
- 幅度缩放、噪声注入；  
- chirp/frame dropout；  
- 对复数输入只做**全局相位旋转**或与天线通道一致的扰动，不要做逐像素随机相位；  
- 对 RD 图可以做背景区域 cutout，但尽量不要把目标主峰随意擦除。  
关于鲁棒性，已有工作显示时间增强、对抗训练和适当的防御策略会明显影响 micro-Doppler 分类稳定性。

对于**领域自适应**，如果训练集和测试集存在明显的**房间差异、距离差异、设备差异或频段差异**，优先考虑：  
- source-only baseline；  
- source-pretrain + target-finetune；  
- DANN/CORAL；  
- synthetic pretrain + real finetune。  
跨频段与无标签目标域的雷达研究都说明，这是比单纯堆深网络更重要的问题。

### 评估指标建议

| 任务 | 主指标 | 补充指标 | 为什么这样评估 |
|---|---|---|---|
| 分类 | Macro-F1、Balanced Accuracy、Confusion Matrix | AUROC、ECE、per-subject F1 | 类别不平衡和被试差异在小样本里很常见，单看 accuracy 往往误导。 |
| 检测 / 跟踪 | mAP / AP、MOTA/MOTP、IDF1 | range/velocity localization error | RadarScenes、RADIal 一类数据集都天然包含检测/跟踪评价目标。 |
| 姿态 | PCK、OKS-mAP、MPJPE | 可见点/遮挡点分开报 | HuPR、MM-Fi、HIBER、MMVR 都提供 keypoints/pose 类标注。 |
| 少样本/迁移 | 1-shot / 5-shot target accuracy | adaptation gain、source-target gap | 小样本论文通常更关心“适配收益”，不是大样本封闭集指标。 |
| 信号处理质量 | SNR 提升、range RMSE、velocity RMSE、track continuity | 空场误警率 | 数据前处理的好坏会直接决定后续学习上限。 |

## 实验方案与最小可行规模

### 分步实验方案

这个方案专门面向“**学生、设备有限、被试有限**”的情况，目标是先把**可复现基线**做出来，再逐步提高复杂度。其逻辑依据是：公开数据集虽然规模不小，但很多成功工作仍然依赖强预处理、合理输入表示和严格数据划分，而不是纯靠大模型。
flowchart LR
A[明确任务与标签] --> B[固定雷达配置与元数据模板]
B --> C[采集空场/标定片段]
C --> D[采集正式样本]
D --> E[生成 RD/RA/点云/轨迹]
E --> F[先做传统基线]
F --> G[再做迁移学习]
G --> H[若样本极少再做少样本/元学习]
H --> I[做 subject-disjoint 与 scene-disjoint 评估]
I --> J[误差分析与补采样]
```

**步骤建议如下：**

1. **只选一个清晰任务**。第一次实验最好只做以下三类之一：  
   - 存在/无人/静止/移动；  
   - 4–6 类全身动作；  
   - 5–8 类动态手势。  
   不建议首轮同时做“姿态 + 动作 + 跟踪 + 多人”。citeturn10search0turn29search0turn22view0

2. **固定一套雷达配置，不要频繁改参数**。  
   小样本时，参数变化会把“类别差异”与“设备差异”混在一起。每次改 chirp 或 frame 配置，都相当于引入一个新域。citeturn42view0turn40search0

3. **先录空场，再录人，再录干扰**。  
   每个场景都要先录背景；每个被试进入前后都留短空窗。这样你后续做静态杂波抑制、归一化和异常帧排查会轻松很多。citeturn36view0turn35view0

4. **第一轮只训练三个基线**：  
   - 特征工程 + SVM；  
   - ResNet-18 微调；  
   - TCN 或 CNN+BiLSTM。  
   只有当这三者都跑通后，才值得引入 ProtoNet、MAML、PointNet++ 或 GNN。citeturn10search0turn18search1turn16search1turn14search0

5. **把“跨人”和“跨场景”拆开评估**。  
   很多学生实验精度高，是因为 train/test 在同一人、同一房间、同一日、同一站位上存在泄漏。最终报告应至少给出：  
   - subject-disjoint；  
   - scene-disjoint；  
   - 同场景随机切分，仅作为上界参考。citeturn24view0turn21view0turn20view0

下表是**工程建议**，不是统一行业标准。它综合了公开数据集中常见的被试规模（约 7–40 人）与迁移/少样本方法的样本效率特点。citeturn25view0turn21view0turn24view0turn29search0turn22view0

| 任务 | 最小可行被试数 | 最小可行场景数 | 每类每人重复次数 | 建议总片段数 | 推荐模型起点 |
|---|---:|---:|---:|---:|---|
| 存在/静止/移动/坐立等 2–4 类任务 | 6–8 | 2 | 20–30 | 300–800 | 特征工程 + SVM；ResNet-18 |
| 4–6 类全身动作 | 8–12 | 2–3 | 15–25 | 800–2,000 | ResNet-18；TCN；CNN+BiLSTM |

### 训练 / 验证 / 测试划分建议

| 被试规模 | 主划分方案 | 适合原因 |
|---|---|---|
| ≤ 10 人 | 外层 **LOSO**，内层从训练被试中再切出验证集 | 小样本下最不容易高估性能 |
| 11–20 人 | **GroupKFold(5)**，按 subject 分组；另做 leave-one-scene-out | 兼顾方差与样本利用 |
| > 20 人 | 60/20/20 或 70/15/15，必须 **subject-disjoint**；再补充 scene-disjoint | 与公开数据集实验更可比 |

**最关键的约束**不是 train/val/test 比例，而是分组键。建议至少按以下层级去重与分组：

`subject_id > session_id > scene_id > recording_day > raw_sequence_id > window_id`

如果同一原始 sequence 被滑窗切成很多片段，这些片段**必须落在同一数据划分里**。否则会产生极强的信息泄漏。citeturn20view0turn21view0turn24view0

### 基线模型实现要点

**基线一：SVM / XGBoost**  
输入用微多普勒或 RD 图的统计特征：Doppler centroid、谱宽、零多普勒能量占比、range dispersion、峰值数、节律周期性。优点是最容易解释，失败也最容易定位原因。citeturn10search0turn10search17

**基线二：ResNet-18 微调**  
输入建议先试三种：  
- 单通道 `log(RD)`；  
- 双通道 `mag + phase_unwrapped`；  
- 双通道 `real + imag`。  
这三种输入的差异值得单独做消融，因为 phase 的价值在 radar 上并不小。citeturn18search1turn37view0

**基线三：TCN 或 CNN+BiLSTM**  
把 16–64 帧组成一个 clip，重叠 50%–75%。如果是连续活动，frame-level 输出比整段 clip-level 输出更有诊断价值。citeturn16search1turn14search0

**点云基线**  
若点云质量稳定，可用 RadHAR 风格的 sliding-window voxel baseline，或者 PointNet++ / Light-PointNet。若点数很少、断裂多，先别急着上图网络。citeturn14search0turn14search11turn17search1turn15search10

## 标注、质量控制与伦理安全

### 数据标注指南

最少应标以下字段：

| 字段 | 说明 |
|---|---|
| `subject_id` | 匿名化编号，不要用真实姓名 |
| `scene_id` | 房间/位置/传感器摆位编号 |
| `action_label` | 主标签 |
| `start_frame`, `end_frame` | 动作起止 |
| `quality_flag` | 正常 / 部分遮挡 / 传感器漂移 / 多人干扰 |
| `config_hash` | 采集参数版本 |
| `sync_anchor` | 若有多模态，同步动作或同步事件编号 |
| `notes` | 衣物/道具/异常行为/站位偏差 |

对**离散动作**，推荐在动作前后各保留 **0.3–0.5 s** 缓冲。  
对**连续动作**，不要只标类别，还要标**边界**。  
对**姿态/跟踪**，要补充可见性、遮挡级别和 track continuity 标记。  
HuPR 说明了利用 synchronized vision-radar 及 cross-modality supervision 来降低手工标注成本；MM-Fi、HIBER、MMVR 则展示了在复杂人感知任务中同时提供动作、位姿、位置或 mask 注释的必要性。citeturn22view0turn24view0turn21view0turn21view3

### 质量控制要点

建议把 QC 看成与模型训练同等重要的流程，而不是附属工作。最关键的检查项如下：

| 检查项 | 通过标准 | 失败后怎么处理 |
|---|---|---|
| 空场基线 | 零多普勒主能量稳定、无异常跳变 | 重新摆位或重录背景 |
| 配置一致性 | 同一实验批次的 chirp/frame/TxRx 配置完全一致 | 分域单独训练或全部重采 |
| 时间连续性 | 无丢帧或 frame index 回退 | 该段数据剔除或单独标记 |
| 同步一致性 | 多模态事件对齐，无明显漂移 | 重新对齐或仅用于单模态 |
| 类别平衡 | 各类样本数差距尽量不超过 2–3 倍 | 重采少数类或做 class-balanced sampling |
| 被试泄漏 | 同一 subject/session 不跨集合 | 重新划分 |
| 场景泄漏 | 同一原始 sequence 的滑窗不跨集合 | 重新划分 |

RADIal 使用逐传感器事件日志，TI 文档明确区分粗同步与硬件同步，公开多模态 benchmark 一般也都对 synchronized frames 和 cross-calibration 有明确要求，因此 QC 的首要对象其实是**时间与配置一致性**。citeturn26view2turn42view0turn22view0turn24view0

### 伦理、隐私与安全注意事项

雷达相对摄像头更有隐私优势，但这不等于“没有隐私风险”。HuPR 明确把 radar 的优势之一表述为在暗光和隐私敏感场景中的适用性；另一方面，雷达也能用于 gait/personnel recognition 或非接触身份识别，因此仍然可能构成可识别的人体行为特征。citeturn22view0turn10search1turn9search3

因此建议：

- **知情同意**：被试必须知道采什么、为什么采、存多久、谁能访问。  
- **最小化原则**：如果相机只用于辅助标注，标注完成后优先保留匿名化结果或直接删除原视频。MM-Fi 官方仓库就明确说明出于隐私原因，原始图像并不完全公开，而改为提供从图像中提取的关键点。citeturn24view1  
- **避免身份化标签**：公开数据时不要保留真实姓名、学号、语音、面部视频和可反查场地信息。  
- **无线电与设备安全**：使用合规开发板与默认发射参数，不私自修改功率前端；多板同步时注意线缆、供电、地线与触发引脚的电气安全。官方文档对 DCA1000、IWR 系列和同步接线均有明确硬件约束。citeturn41view0turn41view1turn42view0  
- **结果表达**：避免把“特定动作识别”泛化叙述成“精神状态、疾病、身份”等高风险推断，除非有单独伦理审批与充分证据。

- 某些数据集的**数据许可**在当前官方页中未明示，尤其是 RADIal；使用前应在下载页或仓库更新中再次确认。citeturn26view2  
- HuPR 虽然是重要 benchmark，但当前官方入口属于**申请访问**而非完全开放直下。citeturn34view3  
- 对 24 GHz 脉冲/低成本 FMCW 模块，我更建议把它们定位成**存在检测、简单手势、微多普勒研究**平台，而不是强行追求角度与高维人体建模；这是基于官方器件通道配置与公开数据集经验做出的工程性判断。citeturn35view6turn41view2turn21view0  
- 像 M4Human 这类最新 benchmark 很有价值，但在当前时间点其站点仍写明“论文发表后发布数据与代码”，所以我没有把它列入“立即可用”的主推荐表。citeturn22view1
